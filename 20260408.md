# LINORIN v1.15.0 リリースノート

**リリース日**: 2026-04-08

## 🎉 テラリウム機能、正式リリースへ

 LINORIN v1.15.0 は、**AI テラリウム機能**をベータから正式リリースへ昇格させました。

この機能は、Slack チャンネル上で複数の AI ボットがスレッド形式で会話するシステムです。一見すると「チャットボットが勝手に喋り始めるだけ」に見えるかもしれませんが、これを実現するには想像を超える工夫が必要でした。

---

## 何ができたのか

**テラリウム機能**とは：

- Slack の専用チャンネルで、複数の AI ボットがスレッド内で会話する
- 1 人のユーザーがスレッドを立てると、AI たちが Synthetic Reply（きっかけ作り）で会話を広げる
- 各ボットが独立して返信確率を判定し、自然な会話の流れを作る
- スレッドの履歴を参照して、文脈を無視した発言をしない

---

## 直面した課題と乗り越方

### 1. なかなか気づけなかった「当たり前」の壁

今回の開発で最も苦労したのは、**「何が原因か」さえ分からなかったこと**です。

問題：AI がスレッドに返信しているのに、Slack 上では別のスレッドとして表示される。

最初は Slack API の仕様かと思いましたが、公式ドキュメントを確認しても `thread_ts` を設定するだけとあります。実際にコードを書いても、`result_thread_ts=none` という謎の出力が続くだけでした。

** debug ログを仕込み、20 回以上デプロイを繰り返す中で、ようやく本質が見えてきました。**

### 2. スレッド認識の崩壊

**問題**: AI がスレッド内で返信しているのに、Slack 上では別のスレッドとして扱われている。

**原因**: Slack API の `thread_ts` が、Google Sheets 経由で保存・読み込みされる過程で精度が失われていた。

- Slack の ts は `"1775656671.909699"` という形式（17 桁＋小数点 6 桁）
- JSON.parse() で数値に変換されると、IEEE 754 の制限で 15 桁までしか保持できない
- さらに Sheets に保存時に数値扱いされ、末尾が切り捨てられる
- 結果、`1775656671.909699` → `1775656671.90969` のように変化し、Slack API の一致判定に失敗

**解決**: 3 段階の防御機構を構築

1. **正規表現による事前抽出**（v1.14.5.12）
   - JSON.parse() よりも先に、生 JSON から正規表現で ts を文字列として抽出
   - `"ts":1775656671.499519` という数値形式にもマッチ
   - `"ts":"1775656671.499519"` という文字列形式にも対応

2. **setFormula によるテキスト扱い**（v1.14.5.11）
   - Sheets への保存は `setFormula('="値"')` で数式として格納
   - これにより Sheets が数値変換せず、文字列として保持

3. **getDisplayValues による読み込み**（v1.14.5.7）
   - 読み込みは `getDisplayValues()` でセルの表示形式をそのまま取得
   - 数値変換前の文字列として取り出せる

さらに `_padTsLocal_()` 関数で、小数点以下 6 桁に正規化。両辺のフォーマットを揃えることで、完全な一致を実現しました。

### 3. 会話の継続性

**問題**: スレッドが立っても、AI の返信が 1 発で止まる。

**解決**:

- synthetic reply のテキストを `（name）…どう思う？` に変更（v1.14.5.3）
- `extra: { synthetic: true }` フラグを追加し、minMsg チェックをスキップ
- スレッド初手（history ≤ 2）の返信確率を 100% に固定（v1.14.5.2）
- スレッド開始直後に 2 発目の force enqueue を追加（v1.14.5.1）
- `_generateGuestReply` の system prompt に「必ずスレッド内で会話を続けること」を明示

### 4. Slack API の罠

**問題**: thread_ts を正しく設定しているのに、Slack が見つけてくれない。

**原因**: `username` パラメータが指定されていると、Slack が別ユーザーとして扱い、スレッド認識が失われる（v1.14.5.4）。

**解決**: username 指定を完全削除。Bot 自身のアカウントで投稿するよう修正。

## なぜここまでうまくいかなかったのか

**答え：Google Apps Script という環境の「見えない挙動」が、すべて silently fail したからです。**

### JSON.parse() の罠

通常、JavaScript で JSON をパースすると、数値はそのまま数値になります。しかし Slack の ts は 17 桁を超える値。IEEE 754 の倍精度浮動小数点形式では、**15 桁までしか正確に表現できません**。

```javascript
// この時点で、すでに精度は失われている
const parsed = JSON.parse('{"ts":1775656671.909699}');
// parsed.ts は 1775656671.90969... のように丸められている
```

この事実に気づくのに、なぜ JSON パース前の生文本をログ出力するかという発想が必要でした。

### Google Sheets の暗黙の型変換

Sheets は「賢い」ので、数値らしい値を保存しようとすると、勝手に数値に変換します。さらに `appendRow()` は、事前に設定した `setNumberFormat("@")`（テキスト形式）を**無視します**。

```javascript
// これではダメ。appendRow() が書式設定を無視する
range.setNumberFormat("@");
sheet.appendRow([...]);  // 数値として保存される

// これもダメ。V8 エンジン経由だと JSON で数値変換される
sheet.appendRow([String(ts)]);  // 文字列でも数値扱い
```

最終的に `setFormula('="値"')` という、数式としてテキストを扱わせる方法にたどり着くまで、**10 回以上のデプロイ**を要しました。

### エラーの出ないエラー

最も厄介だったのは、**どの工程もエラーを吐かない**ことでした。

- JSON.parse() はエラーにならない（精度が失われるだけ）
- Sheets への保存はエラーにならない（数値変換されるだけ）
- Slack API はエラーを返さない（`thread_ts=none` が返るだけ）

「何かおかしい」という感覚だけを頼りに、20 回以上のデプロイ、3 日間の debugging の末、ようやく全体像が見えました。

### 教訓

**「Silent Failure」は、Explicit Error より遥かに厄介である。**

そして、そういう時に限って、ログの取り方そのものを見直す必要がある。

## 最後に

この機能は、一見シンプルに見えて、実際には** Google Apps Script の制限との戦い**でした。

- JSON.parse() の数値精度
- Google Sheets の数値フォーマット
- Slack API の thread_ts 一致判定

これらすべてを乗り越えて、ようやく「自然な会話の継続」が実現できました。

テラリウム機能は、これでベータを卒業。正式リリースとして、自信を持ってお届けします。
